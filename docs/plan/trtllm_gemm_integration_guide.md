# Integration Guide: TRT-LLM Standalone Grouped GEMM (SM100+)

This document covers our key research findings regarding why `attention.rs` grouped GEMMs underperform compared to `sglang` during decoding, and outlines how to integrate flashinfer's `trtllm_batched_gemm` for Linear/MLP layers on **SM100+ (Blackwell)** architectures.

---

## 1. Key Research Findings: Why Decode is Slow in `attention.rs`

Our investigation into the differences between `attention.rs` and the `flashinfer`/`TRT-LLM` backend revealed a critical flaw in how pure CUTLASS is currently being used for decode operations.

### The Autotuning Gap
In `attention.rs/src/kernels/src/fp8_moe_cutlass.cu`, the CUTLASS configuration for Hopper (SM90) is **hardcoded to a single tile shape**: `128x128x128`. 
- **Prefill:** This massive tile size is highly efficient for large token batches.
- **Decode:** In decode, the batch size per expert ($M$) is extremely small (e.g., $M=1$ or $M=4$). Launching a $128$-row CUTLASS MMA block for a $1$-row input forces the GPU to pad the tensor with $127$ rows of zeros. This wastes roughly 99% of the Tensor Cores' compute capacity, tanking memory bandwidth and leading to awful latency.

### The Flashinfer/TRT-LLM Advantage
Flashinfer's grouped GEMMs do not hardcode a single tile size. Instead, they use a **heuristic auto-tuner**.
1.  **Multiple Kernels:** The build system code-generates dozens of different small-M and large-M kernels (e.g., $64\times64$, $64\times128$).
2.  **Runtime Dispatch:** At runtime, the TRT-LLM engine queries the specific $M, N, K$ dimensions and selects the tightly optimized warp-specialized kernel that minimizes padding.
3.  **The Catch for SM90:** We discovered that flashinfer's *standalone* CUTLASS grouped GEMM for SM90 (`group_gemm_sm90.cuh`) **does not actually autotune**. Like `attention.rs`, it hardcodes massive tile shapes. The highly coveted autotuning engine ONLY exists inside the TRT-LLM Codegen binaries.

**Conclusion:** To get the best decode performance, we must use the **TRT-LLM Gen** APIs (`TrtllmGenBatchedGemmRunner`). However, NVIDIA currently only targets the `trtllm_gen` codegen pathway at **SM100+ (Blackwell)** GPUs.

---

## 2. Integrating `trtllm_batched_gemm` for Linear Layers (SM100+)

If the target architecture is SM100+, we can replace the standard `QKVParallelLinear` or standard MoE GEMMs with the `TrtllmGenBatchedGemmRunner`. This will give us auto-tuned, zero-padding decode execution for FP8.

### Step 1: Build System Preparation
Unlike the header-only CUTLASS kernels, the `TrtllmGenBatchedGemmRunner` relies on pre-compiled PTX/SASS cubins generated by `flashinfer`.

Modify `attention.rs/src/kernels/build.rs` to include:
1.  `flashinfer/include/flashinfer/trtllm/batched_gemm/`
2.  Also ensure `flashinfer/csrc/nv_internal/` is included.
3.  Compile `flashinfer/csrc/trtllm_batched_gemm_runner.cu`.

*Note: Flashinfer downloads external cubins for `trtllm_gen` during its own Python `setup.py`. You will need to replicate this download script in `build.rs` to pull the `.cubin` and `flashinferMetaInfo.h` files from the flashinfer object store, or require users to run a python fetch script prior to `cargo build`.*

### Step 2: C FFI Wrapper (`flashinfer_linear_adapter.cu`)

Create a C API to wrap the TRT-LLM C++ class. Since we are targeting simple Linear Grouped GEMM APIs, we don't need the MoE routing features.

```cpp
#include <cuda_runtime.h>
#include <iostream>

// Include the TRT-LLM batched GEMM runner from flashinfer
#include "flashinfer/trtllm/batched_gemm/KernelRunner.h"
#include "flashinfer/trtllm/batched_gemm/trtllmGen_bmm_export/trtllm/gen/DtypeDecl.h"

extern "C" {

// Pointer to an opaque class instance so Rust can hold the state without re-tuning
typedef void* TrtllmGemmHandle;

TrtllmGemmHandle flashinfer_create_trtllm_gemm_sm100(bool is_fp8) {
    using namespace tensorrt_llm::kernels;
    using namespace batchedGemm::trtllm::gen;

    TrtllmGenBatchedGemmRunnerOptions options;
    if (is_fp8) {
        options.dtypeA = Dtype::E4m3;
        options.dtypeB = Dtype::E4m3;
        options.dtypeC = Dtype::Bfloat16; // Typically output to BF16
    } else {
        options.dtypeA = Dtype::Bfloat16;
        options.dtypeB = Dtype::Bfloat16;
        options.dtypeC = Dtype::Bfloat16;
    }
    
    // We are configuring for grouped GEMM (weight layout MajorK)
    options.weightLayout = batchedGemm::gemm::MatrixLayout::MajorK;

    auto* runner = new TrtllmGenBatchedGemmRunner(options);
    return static_cast<TrtllmGemmHandle>(runner);
}

void flashinfer_destroy_trtllm_gemm(TrtllmGemmHandle handle) {
    if (handle) {
        delete static_cast<tensorrt_llm::kernels::TrtllmGenBatchedGemmRunner*>(handle);
    }
}

// Executes an M=1 (or variable M) auto-tuned Grouped GEMM
int flashinfer_run_trtllm_gemm(
    TrtllmGemmHandle handle,
    int32_t num_experts,
    int32_t m, // Total tokens
    int32_t n, // Output dim
    int32_t k, // Input dim
    void* input,
    void* input_scale,
    void* weights,
    void* weights_scale,
    void* output,
    void* workspace,
    cudaStream_t stream
) {
    auto* runner = static_cast<tensorrt_llm::kernels::TrtllmGenBatchedGemmRunner*>(handle);
    
    // 1. Query the auto-tuner for the best heuristic kernel
    int64_t configIndex = runner->getDefaultValidConfigIndex(
        m, n, k, 
        {}, // Empty scale shapes
        m,  // Num tokens
        num_experts, 
        m   // Max num CTAs
    );

    // 2. Execute the grouped GEMM
    int device = 0;
    cudaGetDevice(&device);
    
    runner->run(
        m, n, k,
        {},
        m, num_experts, m, // batch dims
        input, input_scale,
        weights, weights_scale,
        nullptr, nullptr, nullptr, nullptr, // per-token scales 
        nullptr, nullptr, nullptr,          // bias, alpha, beta
        output, nullptr,
        nullptr, nullptr, nullptr, nullptr, nullptr, // permuted indices
        workspace, stream, device, configIndex, true // PDL enabled
    );

    return 0;
}

// Helper to calculate workspace requirements
size_t flashinfer_trtllm_gemm_workspace_size(
    TrtllmGemmHandle handle,
    int32_t num_experts,
    int32_t m, int32_t n, int32_t k
) {
    auto* runner = static_cast<tensorrt_llm::kernels::TrtllmGenBatchedGemmRunner*>(handle);
    int64_t configIndex = runner->getDefaultValidConfigIndex(m,n,k,{},m,num_experts,m);
    return runner->getWorkspaceSizeInBytes(m,n,k,{},m,num_experts,m,configIndex);
}

} // extern "C"
```

### Step 3: Rust Wrapper (`attention.rs/src/linear.rs`)

We then wrap this opaque handle in a Rust struct. The struct should hold the `TrtllmGemmHandle` so that if we ever want to cache tuning states, we don't pay the initialization penalty on every forward pass.

```rust
use candle_core::{Tensor, Result, Error};
use std::ffi::c_void;

extern "C" {
    fn flashinfer_create_trtllm_gemm_sm100(is_fp8: bool) -> *mut c_void;
    fn flashinfer_destroy_trtllm_gemm(handle: *mut c_void);
    fn flashinfer_run_trtllm_gemm(
        handle: *mut c_void, num_experts: i32, m: i32, n: i32, k: i32,
        input: *const c_void, input_scale: *const c_void,
        weights: *const c_void, weights_scale: *const c_void,
        output: *mut c_void, workspace: *mut c_void, stream: cudaStream_t
    ) -> i32;
    fn flashinfer_trtllm_gemm_workspace_size(
        handle: *mut c_void, num_experts: i32, m: i32, n: i32, k: i32
    ) -> usize;
}

pub struct TrtllmGroupedGemm {
    handle: *mut c_void,
    is_fp8: bool,
}

impl TrtllmGroupedGemm {
    pub fn new(is_fp8: bool) -> Self {
        let handle = unsafe { flashinfer_create_trtllm_gemm_sm100(is_fp8) };
        Self { handle, is_fp8 }
    }

    pub fn forward(
        &self,
        input: &Tensor,
        weights: &Tensor,
        input_scale: Option<&Tensor>,
        weights_scale: Option<&Tensor>,
    ) -> Result<Tensor> {
        let (num_experts, k, n) = weights.dims3()?;
        let (m, _) = input.dims2()?;
        
        let output = Tensor::zeros((m, n), candle_core::DType::BF16, input.device())?;
        
        // Ensure you allocate the workspace!
        let ws_size = unsafe { 
            flashinfer_trtllm_gemm_workspace_size(
                self.handle, num_experts as i32, m as i32, n as i32, k as i32
            ) 
        };
        // let workspace = allocate_gpu_workspace(ws_size)?;

        unsafe {
            let stream = get_current_stream(input.device());
            let status = flashinfer_run_trtllm_gemm(
                self.handle, num_experts as i32, m as i32, n as i32, k as i32,
                get_cuda_ptr(input)?,
                input_scale.map(|t| get_cuda_ptr(t).unwrap()).unwrap_or(std::ptr::null()),
                get_cuda_ptr(weights)?,
                weights_scale.map(|t| get_cuda_ptr(t).unwrap()).unwrap_or(std::ptr::null()),
                get_cuda_ptr(&output)?,
                workspace_ptr,
                stream
            );
            
            if status != 0 {
                return Err(Error::Msg("TRT-LLM GEMM failed".to_string()));
            }
        }
        
        Ok(output)
    }
}

impl Drop for TrtllmGroupedGemm {
    fn drop(&mut self) {
        unsafe { flashinfer_destroy_trtllm_gemm(self.handle); }
    }
}
```

### Summary of Deployment Strategy
If you are running on **Hopper (SM90)**: Your best path to auto-tuned decode performance is exclusively through the **Fused MoE pipeline** (`cutlass_fused_moe`), as you cannot run the standalone TRT-LLM GEMM engine on SM90.

If you are running on **Blackwell (SM100+)**: You gain access to the raw TRT-LLM `batched_gemm` executor, allowing you to transparently upgrade all standard MLP/Linear layers with auto-tuned zero-padding kernels using the wrappers defined above.
